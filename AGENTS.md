# Scalius Commerce

## Overview

Turborepo monorepo: TanStack Start admin dashboard + Astro SSR storefront + standalone Hono API, all deployed as Cloudflare Workers. Admin and storefront communicate with the API worker through Cloudflare Service Bindings in production, with HTTP fallbacks for local development.

## Monorepo Structure

```
apps/
  admin-v2/       # @scalius/admin-v2 — TanStack Start admin dashboard Worker
  api/            # @scalius/api — Hono API Worker, queues, cron, WidgetDesignAgent DO
  storefront/     # @scalius/storefront — Astro 6 SSR storefront Worker
packages/
  api-client/     # @scalius/api-client — generated SDK from OpenAPI spec
  core/           # @scalius/core — domain services, auth, providers, integrations, search
  database/       # @scalius/database — Drizzle schema, D1 client, migrations
  shared/         # @scalius/shared — shared utilities
  tsconfig/       # @scalius/tsconfig — shared TS config package
```

## Architecture

### Apps

- **Admin (`apps/admin-v2/`)**: TanStack Start + React 19 admin dashboard. Server functions live in typed domain slices under `src/lib/api-functions/`; query options live in domain modules under `src/lib/api-query-options/`; mutation hooks live in domain modules under `src/lib/api-mutations/`, with `api.mutations.ts` kept only as a compatibility re-export. The broad `api.queries.ts` file has been removed. Use fresh `rg` scans for volatile counts instead of trusting prose. Uses `env.API` service binding in production and Vite proxy/HTTP to `localhost:8787` in local dev. Also has direct D1/KV/R2 bindings for auth, RBAC, and storage initialization.
- **API (`apps/api/`)**: Standalone Hono `OpenAPIHono` app mounted at `/api/v1`. Exports a `WorkerEntrypoint` with `fetch`, `queue`, and scheduled maintenance for orphan inventory expiry, stale hosted-payment cleanup, old/empty abandoned-checkout pruning, and notification-outbox flushing. Owns public/admin API routes, webhook ingestion, OpenAPI spec, queue consumer, and `WidgetDesignAgent` Durable Object for widget AI generation.
- **Storefront (`apps/storefront/`)**: Astro 6 SSR + React 19 customer storefront. Owns product, category, cart, checkout, search, customer auth proxy, SEO, sitemaps, error pages, and L1/L2 caching. Uses `env.BACKEND_API` service binding in production; intentionally skips service binding in local dev because separate Miniflare processes cannot reliably share the Fetcher.

### Packages

- **`@scalius/api-client`**: Generated SDK from OpenAPI. Current generated spec has 259 paths / 357 operations after the checkout/payment-readiness additions; regenerate with `pnpm generate:sdk` after API contract changes instead of trusting prose counts. Uses `@hey-api/openapi-ts` with the bundled Fetch client generator; do not add the deprecated `@hey-api/client-fetch` runtime package back. The generator removes the development-only `/api/v1/media/{key}` passthrough so the generated contract matches production OpenAPI. Do not hand-edit files in `packages/api-client/src/generated/**`.
- **`@scalius/database`**: Drizzle schema and D1 `getDb(env)` client factory. Current schema has 13 schema files and 10 table-defining files; use fresh `rg`/Drizzle scans for volatile table/migration counts instead of trusting prose.
- **`@scalius/core`**: Domain modules in `src/modules/`, Better Auth config, RBAC, providers, integrations, FTS5 search, and cache utilities.
- **`@scalius/shared`**: Shared utilities. It has external runtime deps, but no internal workspace deps.
- **`@scalius/tsconfig`**: Exports `base.json`, `worker.json`, and `astro.json`. Some apps use local framework configs instead of extending it directly.

Packages are JIT-consumed from TypeScript source by Workers/Vite/Astro. Most package manifests expose `.ts` subpaths and do not have build scripts.

## Key Conventions

- **Thin HTTP layer is the target**: Normal `apps/api/src/routes/**` handlers validate/authenticate and delegate to `@scalius/core`. Some webhook and streaming routes still do inline DB work or dispatch because they are integration boundaries.
- **Public listing query ownership**: `/api/v1/products` and `/api/v1/categories/{slug}/products` must keep product SQL in `@scalius/core/modules/products/products.storefront.ts`. Routes may resolve category metadata, validate query params, and call `resolvePublicAttributeFilters()`, but do not reintroduce route-local product predicates, sort SQL, row/count reads, image enrichment, or discount mapping.
- **Product SKU/inventory model**: Products are merchandising containers; SKUs live in `product_variants` and are the only sellable/inventory identities. Simple products use exactly one hidden/default no-option SKU (`isDefault = true`, no size/color); optioned products use merchant-created SKUs with at least one customer option. Do not add product-level inventory or buyer/admin order paths that create new variantless order items. Storefront, quick-buy, COD-only cart submit, and admin manual orders must go through cart/SKU validation or explicit SKU selection. Merchant-created variants/imports/duplicates must not create extra active no-option SKUs, and the hidden simple SKU should be protected from generic duplicate/delete/bulk-edit flows.
- **API route pattern**: New documented JSON endpoints should use `OpenAPIHono`, `createRoute()`, and `app.openapi()`. Exceptions exist for webhooks, health/root/docs/OpenAPI, Partytown proxy, redirects, text responses, and streaming/widget-generation routes.
- **OpenAPI timestamp schemas**: Use shared helpers from `apps/api/src/schemas/timestamps.ts` for mixed string/number/null timestamp fields. Do not hand-roll nullable timestamp unions that regenerate SDK fields as `unknown`.
- **Standardized API errors**: Import API-facing error classes from `apps/api/src/utils/api-error.ts` (re-exported from `@scalius/core/errors`).
- **Standardized API responses**: Use `ok(c, data)`, `created(c, data)`, and `noContent(c)` from `apps/api/src/utils/api-response.ts` for normal JSON endpoints.
- **Response envelope contract**: Normal success JSON endpoints return `{ success: true, data: T }`. The `T` passed to `ok(c, T)` is the final payload; do not nest another `{ success, data }` inside it. Explicit exceptions: root/health/OpenAPI/docs, webhooks, proxies, redirects, text responses, and `204` no-content responses.
- **202 Accepted responses**: Must include top-level `success: true`; use `c.json({ success: true, data: {...} }, 202)` because `ok()` forces 200.
- **Admin API behavior**: Admin server functions in `apps/admin-v2/src/lib/api.server.ts` unwrap API envelopes and return `data` to components. The browser proxy route `apps/admin-v2/src/routes/api/v1/admin/$.ts` passes API responses through unchanged. Admin read-only API calls (`GET`/`HEAD`) use `apps/admin-v2/src/lib/admin-api-timeout.ts` to bound stalled service-binding/HTTP reads through response body consumption; writes intentionally do not get this timeout because a mutation may have committed before the client gives up.
- **Admin dashboard data loading**: The admin home loader must not block SSR or client route entry on dashboard metrics. It skips server-side summary warming, starts `/api/v1/admin/dashboard/home-summary` (`stats` + `recentOrders`, no lifetime `totalRevenue` scan) and `/api/v1/admin/dashboard/activity` as client prefetches, and renders `DashboardSummaryLoading`/chart pending states until real data arrives. Keep `/api/v1/admin/dashboard` and `/api/v1/admin/dashboard/summary` backward-compatible for legacy/full-summary callers; do not make first paint wait on daily activity data again. The chart panel must distinguish `pending`, `success` with empty/all-zero rows, and `error` through `getDashboardActivityPanelState()` so new stores show a settled empty state instead of a permanent skeleton.
- **Admin shell RBAC**: The admin shell uses permission-based RBAC, not legacy `user.role === "admin"`. `apps/admin-v2/src/lib/admin-access.ts` is the pure guard helper; only `/admin/access-denied` remains reachable for authenticated users with no admin permissions. The hot `/admin` guard resolves sessions through `apps/admin-v2/src/lib/admin-session.server.ts`, a narrow D1 lookup for the Better Auth session cookie; do not pull `auth.server.ts`/Better Auth back into `apps/admin-v2/src/lib/auth.fns.ts` for normal route guards. `loadUserPermissions(userId, role, knownIsSuperAdmin)` must keep the known-super-admin fast path before any Cloudflare/database/core RBAC dynamic imports. Non-super-admin RBAC auto-seed uses a versioned six-hour KV marker from `packages/core/src/auth/rbac/auto-seed.ts`; seed definition changes move the marker key, and marker expiry forces periodic D1 truth checks.
- **First-admin setup coordination**: `/api/v1/setup` bypasses normal admin auth only for initial bootstrap, but D1 is the sole authority for concurrency and throttling. `packages/core/src/auth/admin-setup.ts` uses `admin_setup_claims` as a singleton claim ledger and `admin_setup_rate_limits` for the 5/hour/IP throttle. Do not reintroduce Cloudflare KV `get`/`put` locks or KV-only rate limits for setup; KV is eventually consistent and is acceptable only for optional RBAC seed markers after setup.
- **Scanner QR auth coordination**: Scanner QR token minting stays in the admin worker but token consumption is D1-authoritative through `scanner_token_claims` and `packages/core/src/auth/scanner-token-claims.ts`. QR exchange must atomically update an unconsumed, unexpired claim before writing a KV `scanner:session:*` row; never reintroduce KV token `get`/session `put`/token `delete` as the single-use authority. Scanner session payloads should include `claimTokenHash`, and scheduled API maintenance owns bounded cleanup of expired/old consumed claims.
- **Transient D1 overloads**: Use `@scalius/core/utils/transient-d1` for known Cloudflare D1 overload/queue-timeout detection and short read-path retries. Do not duplicate string checks or blindly replay auth/payment/order write paths where the first attempt may have committed.
- **Storefront checkout proxies**: Payment session proxy endpoints unwrap `.data` because the checkout handlers read top-level gateway fields. `apps/storefront/src/pages/api/checkout/create-order.ts` intentionally returns `{ success: true, data: { id } }`, preserves structured `details` on failures, and the browser helper accepts either enveloped or legacy top-level IDs. `apps/storefront/src/pages/api/checkout/validate-cart.ts` is the same-origin proxy for the protected `/api/v1/orders/cart-validation` preflight, uses the shared unsafe-cookie Origin guard, and the browser cart page should use it before COD submit or transfer to `/checkout`. `/buy/{slug}` must also call the storefront `validateCartItems()` helper before writing `quickBuyData` or firing quick-buy analytics; stale product/variant/stock/price/quantity failures redirect to product detail with buyer-safe repair copy.
- **Storefront order creation hot path**: Public checkout order creation requires a stable storefront `checkoutRequestId`, builds the D1 `checkout_attempts` identity, and runs the read-only `resolveExistingCheckoutAttempt()` precheck before mutable checkout policy or IP/phone rate-limit gates. Same-key/same-payload retries replay the committed `201` response or return a pollable `202` while the first request is still processing; same-key/different-payload retries are `409` conflicts. New/non-replay attempts must pass `validateStorefrontCartItems()` before checkout policy, rate limits, or `claimCheckoutAttempt()`; stale product/variant/stock/price issues return one `ValidationError` with `details.itemIssues` and no checkout attempt, KV status, inventory reservation, or payment work is started. Missing, failed, or stale processing attempts must still pass fresh checkout/auth/payment policy and rate limits before `claimCheckoutAttempt()` creates or reclaims the row. Keep the post-claim replay/processing branches for races where another request wins after the read-only lookup. The attempt row reserves the canonical `orderId` and checkout/receipt token so stale or failed attempts can be reclaimed without creating duplicate orders. `createStorefrontOrder()` is the final catalog authority for buyer checkout: it must fresh-read active, non-deleted products, reject inactive/unpublished products, reject variants whose `productId` does not match the submitted item product, resolve variantless simple-product lines only through a real hidden/default or sole no-option SKU, reject optioned products without an explicit selected SKU, and reject SKU-less products. Public checkout commits the D1 order synchronously before returning `201`, then runs COD tracking, order-notification enqueue, and availability cache purge work after commit through `executionCtx.waitUntil()` when available. Discount validation endpoints and app prechecks are advisory; D1 `discount_usage` triggers are the commit-time authority for `maxUses` and one-per-customer coupons, and one-per-customer identity is the immutable `discount_customer_redemptions.customer_key` captured from checkout phone proof, not mutable later `orders.customerPhone` edits. Trigger aborts and late reservation failures must map to buyer-friendly `ValidationError`s with structured details and release any reserved stock before returning. KV checkout status/receipt rows are fast recovery hints; `checkout_attempts` is the fallback authority for `/orders/status/:token` and receipt validation. Do not reintroduce `ORDER_INGEST_QUEUE` or browser polling on the normal buyer path; the storefront 202 polling branch exists only for duplicate in-flight or older async responses.
- **Customer auth policy**: Phone number collection is mandatory platform-wide for customer identity, checkout, delivery, fraud checks, SMS OTP, and WhatsApp OTP. Do not add a merchant setting that disables phone collection. Advanced customer auth policy is stored in `settings.customer_auth/policy` with independent OTP channels (`email`, `sms`, `whatsapp`), email collection mode (not collected/optional/required), and a default OTP channel; `siteSettings.authVerificationMethod` remains only a legacy summary/fallback (`email`, `sms_otp`, `whatsapp_otp`, `both`). Storefront customer auth must keep explicit `sign_in` vs `sign_up` intent: sign-in never creates accounts and should direct unknown customers to create an account, while sign-up rejects duplicate phone/email before mutating OTP challenge state. Customer OTP send throttling is D1-backed in `customer_auth_otp_rate_limits`; `/send-otp` must prefer `CF-Connecting-IP`, trust `X-Forwarded-For` only in loopback local development, hash IP buckets, and treat unknown identity as a shared fail-closed bucket. Customer OTP challenges are D1-backed in `customer_auth_otp_challenges` with channel-scoped keys (`cust_otp:{channel}:{identifier}`), an HMAC-SHA256 code hash, pinned phone/email contact fields, 5-minute expiry, 2-minute resend cooldown, and guarded D1 updates that atomically consume correct codes or increment wrong-code attempts; do not move OTP verification or send-rate authority back to KV. AuthModal passive session hydration must not dispatch `customer-login`; emit it only after an active login/profile completion so account/cart surfaces refresh in place without reload loops. Customer OTP channel policy saves and `/send-otp` runtime must fail closed when a selected provider is not locally ready: SMS readiness requires an active supported SMS provider plus decryptable required credentials, and WhatsApp readiness requires token, phone number id, and template. Admin Auth & Access must save SMS provider credentials before enabling an SMS policy and surface provider readiness blockers beside the selected channels.
- **Checkout payment settings**: Public checkout config must honor `settings.category = "payment_methods"` / `enabled_methods` as the outer storefront gateway allowlist, while still requiring each gateway's own settings to be enabled/configured. Stripe is checkout-usable only when the gateway is enabled and secret key, webhook secret, and publishable key are all present; keep `getStripeSettings()` tolerant for existing server-side payment/webhook/refund work, but never let Stripe enter storefront payment methods or public checkout config without `publishableKey`. SSLCommerz is checkout-usable only when provider-enabled with store ID and store password; Polar is checkout-usable only when provider-enabled with access token, product ID, and webhook secret. Admin `GET /payment-methods` must return raw merchant selection (`enabledMethods`, `defaultMethod`), checkout-flow-filtered buyer-visible selection (`activeMethods`, `activeDefaultMethod`), and per-gateway `configured`/`providerEnabled`/`checkoutSelected`/`checkoutVisible` readiness fields. The saved default method must be selected and visible under the current checkout flow; partial payment and `gateways_only` cannot keep COD as the effective default. Public checkout config may expose `activeDefaultMethod` only when that gateway survives allowlist, checkout-flow, and provider readiness, and the storefront should preselect only a rendered active default. The Payment Gateways UI must keep provider credential saves decoupled from checkout visibility saves; never let a provider switch silently mutate the storefront allowlist or let a checkout visibility switch imply provider readiness. Explicit merchant allowlists fail closed when no selected method is usable; only legacy missing `enabled_methods` defaults to COD. Checkout modes must have compatible usable payment methods before checkout-flow or payment-method/gateway saves can commit: `guest_cod_only` requires COD, `gateways_only` requires at least one enabled/configured online gateway, and `all` requires at least one usable method through the payment allowlist/config rules. Partial payment is an online-payment flow: it requires a positive `siteSettings.partialPaymentAmount`, rejects `guest_cod_only`, hides COD from public checkout config, and must have at least one enabled/configured online gateway before checkout-flow or payment-method/gateway saves can commit. Public checkout config assembly must pass `bypassMemoryCache: true` to payment-method/gateway settings reads because the assembled response is already cached by API KV and storefront L1/L2; never let a stale per-isolate gateway cache repopulate fresh public checkout caches. Payment settings writes (`payment-methods`, Stripe, SSLCommerz, Polar) must invalidate the `checkout` cache group in API KV and purge storefront checkout prefixes (`checkout_config`, shipping/location/checkout-language keys). The admin checkout-flow page uses `queryKeys.settings.checkoutFlow()` even though it reads `/settings/auth`; do not merge it back into the Auth & Access query key, and keep payment-gateway saves invalidating both payment-method and checkout-flow query keys. Gateway credential/settings caches are in-memory only; legacy `gw:*` KV cleanup must be best-effort and must not make an already-committed admin save return a false 500. `/api/v1/checkout/config` assembly/read failures must return non-cacheable `503 CHECKOUT_CONFIG_UNAVAILABLE`; storefront checkout must fail closed with a temporary unavailable state instead of guessing COD availability. Storefront checkout must intersect any session-transferred gateway snapshot with the fresh SSR checkout config, discard empty stale intersections in favor of fresh config, and initial online payment-session calls should send only `orderId` plus receipt proof so the API derives `paymentType`/`depositAmount` from the committed order. Low-value carts must fall back to full online payment instead of creating invalid deposit sessions.
- **Settings/content cache invalidation**: Auth/checkout settings writes must invalidate the `checkout` group because public checkout config is cached. Delivery-provider create/update/delete writes are checkout-affecting and must also invalidate the `checkout` group; the admin delivery-provider UI must invalidate `queryKeys.settings.deliveryProviders()` after successful saves/deletes. CSP/security settings writes must invalidate the `layout` group because `/storefront/csp` and storefront `global_security_settings` are cached; any ancillary CSP KV write must use `getOptionalExecutionContext(c)` or another defensive `waitUntil` guard so local/unit contexts without Hono `ExecutionContext` do not fail committed writes. Category and CMS page writes must also invalidate `layout` because storefront fallback navigation is derived from public categories/pages when no explicit header navigation config exists. Committed admin writes should await API KV invalidation, then schedule the storefront purge with `executionCtx.waitUntil()` so a downstream purge/network failure cannot turn an already-committed mutation into a false 500. Use `invalidateCatalogCaches(domain, c, { htmlPaths })` for catalog writes, `invalidateApiAndScheduleStorefrontGroups(groups, c, { htmlPaths })` for settings/content/reference writes that need exact warm paths, and `triggerStorefrontPurgeForPrefixes(..., getOptionalExecutionContext(c))` for widget exact-prefix purges after API pattern invalidation. Product and discount catalog writes warm canonical `/search`; product create/update/delete/restore/permanent-delete and variant writes must also pass affected `/products/{slug}` exact paths before category/listing paths; category catalog writes warm canonical `/search` plus affected `/categories/{slug}` paths; product writes should pass old/new affected category paths when the route already has or can cheaply resolve them. CMS page create/update/bulk-publish/restore/single-restore should pass post-write public `/{slug}` exact paths through `invalidateApiAndScheduleStorefrontGroups(["pages", "layout"], c, { htmlPaths })`; delete/unpublish paths keep freshness invalidation but should not rewarm now-hidden pages. Attribute writes use `invalidateApiAndScheduleStorefrontGroups(["attributes", "products"], c, { htmlPaths: ["/search"] })` because public filter sidebars derive from attribute assignments and search facets. Product catalog writes must invalidate the `attributes` group too because public filter sidebars derive from product category/attribute assignments. Scoped widget purges for product/category/page/collection placements must include exact rendered `htmlPaths`; widget writes must also resolve published CMS pages containing matching widget shortcodes and purge those exact page render/HTML paths. Only homepage/global widget changes should bump the global storefront version and warm the homepage, except shortcode scans that exceed the exact path cap or fail after a committed write may bump globally for freshness. Storefront exact `htmlPaths` are relative-only, canonicalized with the same HTML cache-key rules before dedupe/capping, capped by `MAX_STOREFRONT_EXACT_HTML_PATHS` / `MAX_EXACT_HTML_WARM_PATHS` (`20`), and the storefront warms them in small batches so `waitUntil()` work remains bounded. Stock-changing order/payment/delivery/cron paths should use `invalidateProductAvailabilityCaches()` or preload with `resolveProductAvailabilityCacheSubjects()` before destructive order item writes, then call `invalidateProductAvailabilityCacheSubjects(subjects, c, db)` after commit; this clears exact product detail/search API KV and sends storefront exact data keys plus `/products/${slug}` and any CMS pages containing matching product shortcodes without broad catalog busting. Admin inventory stock-only routes (`/{variantId}/adjust`, `/stock-adjust`, `/stock-set`) should invalidate by affected `variantId` through the product availability helper, not broad catalog groups. Use `invalidateApiAndStorefrontGroups()` or `purgeStorefrontForGroups()` only when the route intentionally needs immediate storefront purge success before responding. `CACHE_TTLS.NONE` is a true cache bypass; do not rely on downstream KV TTL clamping for no-cache routes.
- **Public hero cache variance**: `/api/v1/hero/sliders` may use API KV cache only when callers pass an explicit `type=desktop` or `type=mobile`; it must vary by query. Untyped slider-list requests are user-agent derived and must bypass cache, otherwise mobile and desktop slider payloads can cross-serve. Slider detail reads that are not the list endpoint may stay normally cached.
- **API query cache keys**: API `cacheMiddleware({ varyByQuery: true })` canonicalizes query params before building KV keys. Public routes with schema defaults should pass `queryDefaults` so default query strings do not create duplicate cache entries, but use path-aware defaults when a router has multiple routes with different defaults.
- **API KV cache fences**: `cacheMiddleware()` stores API response bodies under versioned physical keys from `apps/api/src/utils/api-cache-fence.ts`; invalidators must bump the matching fence scope before deleting old keys so in-flight `waitUntil()` miss writes cannot resurrect stale data. Group/pattern invalidation should use `bumpApiCacheFences()` before `deleteCacheByPattern()`, and exact no-query keys such as product details should use `deleteVersionedCacheKeyFamily()` instead of broad prefix deletion. Do not reintroduce `_last_cleared:*`; `/api/v1/cache/last-cleared` reads fence `updatedAt` values.
- **Storefront cache namespace, query keys, and generations**: Storefront KV version keys and exact/data generation keys must use the canonical cache namespace from `CACHE_NAMESPACE`, then `STOREFRONT_URL` hostname, then request hostname. Cache API keys still use the actual request hostname/origin so local/staging hostnames work. HTML cache keys and product-list L2 keys must use `apps/storefront/src/lib/cache-key.ts` plus `apps/storefront/src/lib/product-list-query.ts` so surviving query params are deterministic, repeated render-affecting params collapse to the last surviving value before redirects/cache lookup, tracking/unknown params do not fragment HTML/L2 caches, product client-side `size`/`color` selections do not create separate product HTML entries, and category/search dynamic attribute filters are kept only when the current filter metadata proves the key/value is valid. Exact/data purges should read the old generation before bumping, delete old generation-specific L2/HTML keys best-effort, bump KV generations for freshness, and warm affected HTML paths after canonicalizing them through `@scalius/shared/storefront-cache-path` so noisy query variants do not consume the bounded warm budget. Checkout-only data purges are generation-scoped (`checkout_config`, `global_checkout_language`, `global_shipping_cities`, `global_shipping_methods`, `shipping_zones_`, `shipping_areas_`) and must not bump the shared HTML/global version or warm `/`; unknown prefix-only purges still bump globally as the safe fallback.
- **Storefront inline JSON**: Never interpolate raw `JSON.stringify()` into executable inline scripts, JSON-LD script tags, `type="application/json"` data islands, or generated HTML strings that include `<script>`. Use `serializeJsonForInlineScript()` from `apps/storefront/src/lib/safe-json.ts` for any JSON value assigned through `set:html`/`is:inline`, and cover `</script>` breakout payloads with DOM-parser and source-boundary tests.
- **Credentialed API CORS and cookie Origin guards**: Merchant CSP/security domains (`settings.security/csp_allowed_domains` and `security:csp_allowed_domains`) are CSP-only and must never expand credentialed API CORS. `getCorsOriginContext()` allows exact first-party runtime origins plus explicit exact URL origins from credentialed-CORS env vars, with no wildcard env entries, URL credentials, or CSP/KV reads. Loopback wildcard origins (`localhost`, `127.0.0.1`, `[::1]`) are allowed only when a first-party runtime URL is itself loopback; production first-party URLs must not implicitly allow local origins. Unsafe cookie-bearing browser requests to API/admin/customer-auth/order surfaces must pass `cookieOriginGuardMiddleware` or the shared same-origin proxy guard before route/auth/session mutation logic runs; service-binding/server-to-server calls without a browser `Origin` continue through normal route auth.

- **Payment session proof and policy**: Stripe, SSLCommerz, and Polar session creation must require the order receipt token, validate it against `order_receipt:{token}` before gateway settings/provider calls, and derive callback/success/cancel URLs from trusted runtime config such as `PUBLIC_API_BASE_URL`, not request-body URLs. Payment amount/type/currency/capture behavior must be resolved through `apps/api/src/routes/payment/payment-session-policy.ts` and server settings/order state: reject soft-deleted, cancelled, returned, refunded, partially-refunded, fully-paid, or payment-refunded orders before gateway settings/provider calls; derive omitted `paymentType` from the committed order total plus current partial-payment settings; reject deposits when partial payment is disabled, a requested deposit does not match configured `siteSettings.partialPaymentAmount`, money has already been collected, or an existing plan is not pending with matching amounts; allow `balance` only when `orders.paymentStatus = partial`, `paidAmount > 0`, and `payment_plans.status = deposit_paid` with matching balance; ignore caller currency for session creation; keep public Stripe sessions `manualCapture: false`; and generate a unique SSLCommerz merchant `tran_id` per full/deposit/balance payment attempt while carrying the canonical order id separately in `value_b`/trusted callback `order_id`. Deposit payment-session routes must persist the pending `payment_plans` row before any provider call and fail before provider calls if that write fails. Session creation must fresh-read the outer payment-method allowlist and gateway credentials with `FRESH_GATEWAY_SETTINGS_READ_OPTIONS`, so stale checkout tabs cannot create new external payment sessions after a merchant disables or rotates a gateway.

Settings-read failures must be retryable 5xx responses and must not claim an event; invalid signatures/IPN validation failures must not claim an event. Duplicates return success without enqueueing; queue failures mark the event `failed` and return retryable `503`. Fresh `processing` claims dedupe, stale `processing` claims are lease-reclaimable, and `queued`/`processed` claims remain terminal. SSLCommerz IPN must validate `val_id` first, use canonical validated `tran_id`/`val_id`/payment fields, resolve the canonical order id from validated `value_b` or parsed scoped `tran_id`, and resolve payment type against server-side order/payment-plan state instead of trusting form metadata such as `tran_id` or `value_a`.
- **Payment refund provider freshness**: Admin refund and auto-return refund dispatch must fresh-read Stripe/SSLCommerz/Polar gateway settings with `FRESH_GATEWAY_SETTINGS_READ_OPTIONS` after the guarded local refund claim and before the external provider call. If fresh settings are missing, disabled, or unreadable, `processRefund()` must release the local refund claim and restore the previous order payment state before surfacing the error.
- **Payment event consumers**: Confirmed gateway events use `processPaymentConfirmed()` and gateway-ID partial unique indexes. `processPaymentConfirmed()` must reject soft-deleted, cancelled, returned, refunded, partially-refunded, fully-paid, or payment-refunded orders before creating/promoting a local payment claim, validate the incoming leg (`full`, `deposit`, or `balance`) against order amount/payment-plan state, and include the payable-state plus matching payment-plan predicates in the final CAS/batch success contract. Deposit plan updates only move pending matching plans to `deposit_paid`; balance plan updates only move `deposit_paid` plans to `completed`, and a zero-row plan update is not a successful payment application. SSLCommerz confirmed-payment idempotency is by canonical `sslcommerzValId`; `sslcommerzTranId` is an attempt/correlation field and cannot be the split-payment uniqueness key. These non-payable late successes are non-retryable queue outcomes for manual reconciliation; they must not mutate order totals/status. Failed gateway events must be idempotent and must not block a later successful event for the same retryable intent. Payment cancellation inventory release goes through `applyInventoryForStatusChange(..., OrderStatus.CANCELLED)`. Polar webhook refunds must CAS-update payment and order status through refund state-machine rules before releasing pre-fulfillment inventory.

The API cron flushes pending/failed/stale claimed rows every 15 minutes. Admin FCM permanent stale-token responses include exact registration-token codes plus Firebase variants such as `Device unregistered`/`NotRegistered`; treat them as skipped receipts and deactivate the token only after the receipt is terminal, while preserving transient `unknown-error`/server failures as retryable. Resend receives the receipt key as its `Idempotency-Key`; GenNet receives a deterministic `csms_id`; Cloudflare Email, Meta WhatsApp, and FCM store provider receipts but do not expose a first-class upstream idempotency key, so a crash after provider acceptance but before receipt persistence can still duplicate on those providers.

- **Inventory expiry**: `releaseExpiredReservations()` is for orphaned reservation movements whose order row is missing. Do not use that orphan-reservation sweeper to cancel stale existing orders; existing orders must move inventory through explicit order transition logic so `orders.inventoryAction`, `productVariants`, and movement logs stay consistent. The sweep is intentionally bounded (`limit` default `50`, max `200`) and returns `hasMore` for backlog visibility. Expiry releases use deterministic movement ids inside a D1 `safeBatch()` claim+counter update so overlapping cron runs cannot double-apply the same orphan release; do not replace that with best-effort random movement logging.
- **Abandoned-checkout/stale hosted-payment cleanup**: Existing stale hosted-payment orders are handled only by `archiveStaleIncompleteOrders()` in `@scalius/core/modules/orders/stale-incomplete-orders`. It targets `stripe`/`sslcommerz`/`polar` only, with `status = incomplete`, `paymentStatus IN (unpaid, failed)`, `paidAmount <= 0`, `deletedAt IS NULL`, no active shipment claim, no pending/succeeded `order_payments`, no live `payment_session_attempts` processing lease, and a matching stale cutoff. Orders with `inventoryAction` `reserved` or `deducted` then release/restore through `applyInventoryForStatusChange(db, orderId, OrderStatus.CANCELLED)`. If release fails, roll the cleanup claim back to `incomplete`; do not archive or soft-delete. The final archive must be guarded, pending payment-plan cancellation must be fenced on the order actually being finalized, and the `abandoned_checkouts` archive row is inserted only after finalization. Old/empty abandoned-checkout row pruning lives in the bounded `cleanupStaleAbandonedCheckouts()` scheduled helper; admin `GET /api/v1/admin/abandoned-checkouts` must remain side-effect-free because view-only admins can call it. Never hard-delete stale incomplete orders before inventory release succeeds.
- **Order status inventory reconciliation**: Status, fulfillment, COD, delivery, refund, and return paths must call `applyInventoryForStatusChange()` even when retry sees the order already at the requested/mapped terminal status. Same-status retries must reconcile stale `orders.inventoryAction`; do not return early before inventory repair. Full-refund retries may release pre-fulfillment reservations for already-cancelled orders, but must not auto-restore fulfilled/deducted inventory.
- **Order inventory transition idempotency**: `applyInventoryForStatusChange()` owns `reserved -> deducted/restored`, `deducted -> restored`, and `restored -> reserved` transitions. Deduct/release/restore transitions must use deterministic, movement-generation-based `inventory_movements.id` claims (`transition:*`) plus the variant `stockVersion` CAS update in one D1 `safeBatch()`; exact duplicate claims are idempotent retries, mismatched claims fail closed for manual reconciliation, and the final `orders.inventoryAction` update must be CAS-guarded against the action observed before the stock transition. Do not key these claims on `orders.version`, because unrelated order edits must not make a crash retry double-apply stock. Do not reintroduce the old best-effort `deductMultiple` / `releaseMultiple` / `restoreDeductedMultiple` path for order status reconciliation.
- **COD payment state**: Generic order status updates must never synthesize COD payment state. COD collection must go through `processCodAction(..., { action: "collected" })` / `recordCODCollection()` so `cod_tracking`, `order_payments`, `orders.paymentStatus`, `orders.paidAmount`, and `orders.balanceDue` move together. Generic COD `delivered`/`completed` transitions may proceed only when successful COD payment and collected COD tracking evidence already exist and the order is fully paid.
- **Admin full order edits**: `updateOrder()` must preserve old item context until inventory deltas are safely applied. Positive deltas are reserved/deducted before the order CAS; negative reserved/deducted deltas and terminal release/restore are also applied before replacing `order_items`. Delta failures must reject the edit, not log-and-succeed. Item replacement uses one D1 batch so insert failure cannot strand the order with deleted old items, and pre-write inventory compensation must run when later writes fail.
- **Trash restore inventory policy**: `restoreOrder()` clears `deletedAt`; it is not a hidden order-status transition. Use the shared reservable-status policy from `isStockReservableStatus()`: `incomplete`, `pending`, `processing`, and `confirmed` may re-reserve stock and become `inventoryAction = "reserved"` when variant items exist, or `none` when no variant inventory exists. `cancelled`, `returned`, and `refunded` remain `restored`; shipped/delivered/completed/partially-refunded restored orders must reject until inventory/status are explicitly reconciled. Existing `reserved` and `deducted` actions are allowed only with compatible statuses, and a successful re-reservation must be released if the final restore CAS fails.
- **Shipment creation**: Route-facing provider shipment creation must go through `@scalius/core/modules/orders` fulfillment helpers. Bulk/provider shipment creation owns an order-level `shipmentClaimId` / `shipmentClaimExpiresAt` lock linked to the insert-first `delivery_shipments` row. Admin order/status/COD/fulfillment/edit/delete/refund/payment-session/shipment-refresh paths must reject active claims; cleanup should skip them; queue/webhook paths must retry rather than acknowledge skipped external truth. Provider success with failed local finalization marks the shipment `reconcile_required` and keeps the order claim active for reconciliation. `packages/core/src/modules/delivery/delivery.service.ts#createShipment()` is a low-level primitive and assumes the caller has already performed the durable order claim/transition.
- **Manual fulfillment**: `createFulfillmentShipment()` must first acquire a private `shipmentClaimId` without publishing `status = shipped` or `fulfillmentStatus = complete`. The manual `delivery_shipments` insert, scoped `order_items` updates, and final order status/fulfillment/claim-clear update must be committed together in one D1 batch. Fulfillment `itemIds` must be non-empty, unique, and belong to the same order; item updates must include `orderId` in the predicate. If the batch fails before the shipment row exists, clear the private claim and leave inventory untouched.
- **Shipment deletion**: Delete shipments only through `deleteShipmentRecord()`. It must reject active order shipment claims, `reconcile_required` rows, and unresolved expired matching claims; only failed/cancelled stale claimed shipments may clear the old order claim before deletion.
- **Order ingest queue isolation**: `handleOrderIngestBatch()` may batch DB writes for throughput, but stock reservation, DB-write fallback, checkout status, ack/retry, and rollback decisions must remain per order. A rejected/acked message must not be retried because another message in the same queue batch failed. Before reserving, classify active `inventory_movements` for the order: exact active reservations are reused for crash/redelivery replay, missing entries are reserved with `reserveStockBatch(..., { reservationKey: "checkout-ingest:v1" })`, and quantity/stray movement mismatches fail closed for manual reconciliation. After a shared DB batch failure, re-check whether each order row committed before mutating inventory, reuse any reservation already acquired for isolated replay, and only retry after that order's known reservations are confirmed released. If release is uncertain or `releaseMultiple()` reports failure, fail the checkout closed for manual reconciliation instead of redelivering into a second reservation.
- **Batch reservation claims**: `reserveStockBatch()` writes reservation movement claims and variant counter updates in the same D1 `safeBatch()`. Movement inserts are gated by the pre-read `stockVersion`; empty movement/update returns are rolled back and retried. Deterministic `reservationKey`/`movementId` claims are opt-in for replay-sensitive flows, exact duplicate deterministic claims are idempotent success, and mismatches must surface `manualReconciliationRequired` rather than silently reserving again. Keep `inventory_movements.order_id` non-FK-enforced because checkout reservation claims are written before the queued order row commits.
- **Order ingest legacy queue**: `ORDER_INGEST_QUEUE` remains configured for legacy/internal queued ingest handlers, but it is not on the normal storefront checkout path. If async ingest is intentionally reintroduced, status and receipt proof must be D1/checkout-intent authoritative rather than KV-only, and browser polling must not block payment handoff on best-effort queue state.
- **Storefront shipping verification**: Storefront order creation must require a valid active, non-deleted `shippingMethods` row whenever shipping applies and derive `shippingCharge` from that method fee. Browser-provided `shippingCharge` is not authoritative. Free-delivery products may explicitly waive shipping method and charge.
- **Delivery webhook/status semantics**: Pathao and Steadfast webhooks must claim durable lease-backed `webhook_events` before shipment/order side effects. Delivery event identities must include enough provider event/status/update data to allow later status changes for the same shipment; Steadfast delivery-status identities include the raw status. Delivery webhooks and refresh routes should call `updateOrderStatusFromShipment()` even when the shipment row already has the provider status so failed prior attempts can reconcile order inventory; customer notifications still require an actual order status change. Any status emitted by `packages/core/src/modules/delivery/status-mapper.ts` should either be handled explicitly in `updateOrderStatusFromShipment()` or documented as shipment-only. `shipped -> confirmed` is allowed only for failed carrier delivery retry semantics.

- **Database migration metadata**: `packages/database/scripts/check-migration-metadata.mjs` enforces SQL/journal/snapshot consistency and documents the allowed manual migrations without snapshots. Update the allowlist only when intentionally adding a manual migration without a Drizzle snapshot.

- **Storefront build ID**: `apps/storefront/src/config/build-id.ts` is generated and git-ignored. `apps/storefront/scripts/generate-build-id.js` must run before storefront typecheck/build/deploy. It prefers commit SHA env vars and otherwise hashes stable source/config inputs; do not reintroduce timestamp-only build IDs.

Keep each app's `env.d.ts`/`hono-env.d.ts` in sync and run `pnpm check:env` after binding/var changes. ## Storefront Notes

- Runtime Cloudflare env is made request-scoped via `apiContext`/AsyncLocalStorage in `apps/storefront/src/middleware.ts`; API base URL resolution is lazy in `apps/storefront/src/lib/api/client.ts`. Browser API calls require configured `PUBLIC_API_URL`/injected `window.__API_BASE_URL__`; the storefront intentionally does not fall back to a fake same-origin `/api/v1` proxy.
- Storefront listing/detail pages must preserve their cold-cache promise boundaries. Product detail starts layout and product reads together, then chains product-scoped widgets from the product promise so widgets do not wait for layout. Search starts layout, `getAllProducts(productListOptions)`, and search filter metadata together after URL options are built when there are no dynamic attribute params. Category pages start layout, `getProductsByCategory(slug, productListOptions)`, filter metadata, and category-widget reads together on the normal path; they must use the full category object returned by `/api/v1/categories/{slug}/products` and must not issue a separate `getCategoryBySlug()` read on the normal path. Category/search URLs with candidate dynamic attribute params must validate through `resolveProductListQueryState()` after scoped filter metadata is available, redirect noisy/invalid key-value pairs to the canonical URL, then fetch the product list with only validated filters. Category/product widgets may chain from the entity promise because they require the entity id. CMS pages must trust `getPageRenderData(slug).widgets` and must not refetch scoped page widgets when the render endpoint returns none. Category/search page URL query `q` must be normalized with `normalizeSearchQuery()`, category `q` must map to API `search`, and `q` must not be forwarded as an attribute filter. Keep `apps/storefront/src/lib/page-data-boundaries.test.ts`, `apps/storefront/src/lib/product-list-query.test.ts`, `apps/api/src/routes/categories-boundaries.test.ts`, and `packages/core/src/modules/products/products.storefront-boundaries.test.ts` aligned with these latency contracts until consolidated render-data endpoints replace the separate calls.
- API `cacheMiddleware` reads KV before route handlers, but cache-miss writes must use `executionCtx.waitUntil()` when available and fall back to awaiting only in local/test contexts without a Hono execution context. Do not move KV writes back onto the response path for public product/category/layout/attribute reads.
- Same-origin customer auth must go through `apps/storefront/src/pages/api/customer-auth/[...path].ts` so `Set-Cookie` works on the storefront domain. Logout clears host-only cookies in `apps/storefront/src/pages/api/auth/logout.ts`.
- Customer account GETs (`/api/v1/customer-auth/me` and `/api/v1/customer-auth/orders`) return private customer/order data and must keep explicit `Cache-Control: private, no-cache, no-store, must-revalidate` headers. Storefront callers should also request them with `cache: "no-store"`.
- HTML caching is allowlisted in middleware. Cart, checkout, account, health, API routes, private sessions, and payment-sensitive flows must bypass cache. Cacheable HTML URLs should be canonicalized through `buildHtmlCacheBaseUrl()` before adding `cache_v`/`cache_gen`; do not hand-build equivalent Cache API URLs in middleware or purge deletion paths.
- Product, category, collection detail, search, and CMS-page HTML are cacheable through `apps/storefront/src/middleware.ts`. The same Cache API lane also covers explicit generated public XML/XSLT/text assets (`/sitemap.xml`, `/sitemap-*.xml`, `/sitemap.xsl`, `/robots.txt`, and `/api/facebook-feed.xml`) through `isCacheablePublicResponse()`, while JSON/API/private/payment-sensitive responses must stay excluded by path, content type, `Set-Cookie`, and `Cache-Control` guards. Collection detail must stay in `CACHEABLE_PATHS` because collection-scoped widget purges already send exact `/collections/{id}` `htmlPaths` and `htmlPathCacheKeyFromPath()` already assigns exact generations to those paths.
- Storefront `withEdgeCache()` must keep in-flight request dedupe even when KV cache-version lookup is unavailable, but it must not populate L1/L2 storage without a cache version. KV/version failures should be an origin-gentle cache bypass, not a stale-data fallback.
- Scoped widget purges that include exact `htmlPaths` must not bump the global storefront cache version just because they also clear L1/data prefixes. Bump exact generations for `widget_*`, `widgets_scope_*`, `page_render_*_${BUILD_ID}`, and exact entity HTML paths (`/products/*`, `/categories/*`, `/collections/*`, CMS slugs), then warm the supplied paths. Prefix-only purges for unknown/non-generation-scoped prefixes still bump the global version because L2 Cache API cannot delete by prefix.
- `/api/purge-cache` is mutating only through `POST`. `GET` must stay non-mutating and return `405 Allow: POST` except for query-string credential rejection. Purge credentials in query strings are rejected; send `Authorization: Bearer ...` or the configured purge-token header. Full POST purges bump the KV cache version and clear L1. Selective unknown prefix POST purges also bump the KV cache version so L2 Cache API keys move; whenever the global version moves, critical pages must be warmed immediately. Checkout-only selective purges are data-generation scoped and must bump the relevant `g:{namespace}:...` keys without moving `v_{namespace}` or warming `/`. Exact product purges must bump per-key KV generations for `product_slug_*` / `product_variants_*`, then may best-effort delete local Cache API keys and warm affected product paths; do not rely on `cache.delete()` alone for global freshness. Cache warming must use `url.origin` so local/staging ports are preserved.
- CMS shortcode parsing must tolerate sanitized attribute quote entities (`&quot;`, `&#34;`, `&apos;`, and numeric variants) because rich content is sanitized before expansion. Product/widget shortcode resolution should dedupe by shortcode type and id/slug inside a single render so repeated placeholders do not fan out duplicate API reads, while still replacing every occurrence in the rendered HTML.
- Public CMS page visibility is centralized in `packages/core/src/modules/pages/pages.service.ts`; public routes and page sitemaps must enforce `isPublished`, not deleted, and `publishedAt` null or not in the future.
- Public collection detail product resolution is centralized in `packages/core/src/modules/collections/collections.service.ts`. Manual `productIds` and configured `categoryIds` must be mapped back into the merchant-configured order after DB reads because SQL `IN (...)` result order is not stable; missing/inactive products may be skipped, but later visible products should still fill up to `maxProducts`.
- Cart supports COD server POST and multi-gateway sessionStorage redirect to `/checkout`; checkout gateway handlers live in `apps/storefront/src/lib/checkout/`. Preserve cart contents for external-gateway cancel/failure recovery, but clear raw checkout transfer state as soon as the browser no longer needs it. Completed-order redirects clear checkout state and cart; SSLCommerz/Polar external gateway redirects clear checkout session state after order/session creation while leaving cart intact. If SSLCommerz/Polar session setup fails after the order commits, redirect to receipt-token recovery with `result=failed`; Stripe card/session failures should stay on checkout until receipt-page Stripe retry exists, because `/order-success` retry controls are hosted-gateway-only.
- Storefront Meta CAPI browser dispatch may include only browser identifiers such as `_fbp`, `_fbc`, and user agent by default. Do not persist checkout/customer PII in standalone `sessionStorage` analytics keys or auto-enrich broad events (`ViewContent`, `Search`, `AddToCart`, `InitiateCheckout`, `AddPaymentInfo`) from checkout state. Any PII in `userData` must be explicitly supplied for a narrow conversion event and reviewed with the relevant privacy/consent context. Pixel/CAPI paired events must share the same browser event id: Pixel gets Meta's `eventID` option and `/api/v1/meta/events` gets `eventId`. Product and purchase CAPI dispatch must not be gated on `window.fbq`; CAPI still needs to send when Pixel is blocked.
- Meta Graph API integrations must use `META_GRAPH_API_VERSION` from `packages/core/src/integrations/meta/conversions-api.ts`; do not hardcode old versions in CAPI or WhatsApp senders. As of 2026-06-19 this is `v25.0`. Meta CAPI settings are analytics-scoped (`ANALYTICS_VIEW`/`ANALYTICS_EDIT`), new CAPI access-token saves are encrypted with `CREDENTIAL_ENCRYPTION_KEY`, settings writes invalidate the `layout` cache group, public `/api/v1/meta/events` validates `eventSourceUrl` against `STOREFRONT_URL` and rate-limits through API KV when present, and CAPI request logs must redact `user_data`, `test_event_code`, and event-source URL queries before storage/display. `/order-success` purchase tracking is guarded by `claimMetaPurchaseTracking(order.id)` so browser reloads do not replay the same Purchase event. WhatsApp OTP and order notification senders share the same Graph API version and resolve Meta Cloud API credentials through `getWhatsAppCloudApiSettings()`: the access token is stored encrypted at `settings.whatsapp/access_token` with `CREDENTIAL_ENCRYPTION_KEY`, while `site_settings.whatsapp_access_token` is legacy fallback only. WhatsApp legacy migration and cleanup must pass a dedicated `migrationEncryptionKey` from `getCredentialEncryptionKey()`; `getEncryptionKey()` may be used only for tolerant reads. The WhatsApp phone-number ID and auth template name remain non-secret `site_settings` fields. Auth OTP queue payloads must never include Meta tokens, phone-number IDs, or template settings; the API queue consumer resolves credentials at send time. ## How-To Recipes

### Add a New Field to an Entity

1. Schema: `packages/database/src/schema/{domain}.ts` — add column.
2. Migration: `pnpm db:generate`.
3. Validation: `packages/core/src/modules/{domain}/{domain}.validation.ts` — update Zod schemas.
4. Service: `packages/core/src/modules/{domain}/{domain}.admin.ts` or equivalent service — update selects/inserts/updates.
5. API route: usually no change if it already passes validated service data through; otherwise update route schemas.
6. Admin form schema/component: update the relevant form files in `apps/admin-v2/src/components/admin/`.
7. Storefront display: update storefront-facing core service/API client/storefront UI as needed.
8. SDK: run `pnpm generate:sdk` if the OpenAPI surface changed.

### Add a New Payment Gateway

See `packages/core/src/modules/payments/README.md`. Current payment reality is mixed: legacy payment modules/gateway registry are active, and the universal provider registry has a Stripe adapter. Coordinate storage, admin settings, webhook routes, queue events, SDK regeneration, and tests.

### Add a Notification Type

1. Add the key and label to `packages/core/src/modules/notifications/notification-types.ts`.
2. Add template/channel handling in `packages/core/src/modules/notifications/notifications.service.ts`.
3. Confirm defaults and validation in `packages/core/src/modules/settings/settings.service.ts` still merge the new type.
4. Queue message typing in `apps/api/src/queue-consumer.ts` should pick it up through `OrderNotificationType`.
5. Trigger by enqueueing `{ type: "order.notification", ... }` to `ORDER_NOTIFICATIONS_QUEUE` from the route/service that owns the state transition.
6. Update admin notification settings UI only if the shared labels/ordering are not sufficient.

### Add a New Admin Settings Tab

1. Component: create `apps/admin-v2/src/components/admin/settings/MySettingsBuilder.tsx`.
2. Route: add sub-route under `apps/admin-v2/src/routes/admin/settings/`.
3. API: create or extend `apps/api/src/routes/admin/settings/*`.
4. Storage: use `siteSettings` for typed singleton fields and `settings` for provider/category config.
5. Add query/mutation wrappers and invalidate the right query keys.

- `pnpm dev:api`, `pnpm dev:admin`, and `pnpm dev:storefront` all run through `scripts/dev.sh`, apply pending local D1 migrations unless skipped, and wait for API readiness before starting dependent apps.

## Known Backlog / Limitations

- **Active audit backlog**: Check `audit/REMEDIATION_TRACKER.md` before choosing the next remediation slice. The prior live storefront missing-CDN-object reference was fixed through the admin product update path and verified against API data, D1 rows, CDN response, and storefront HTML.
- **Mixed provider systems**: Universal provider registry currently has Stripe payment + Resend email adapters, while transactional email runtime selection now uses Cloudflare Email Service by default with Resend fallback. SMS still uses the legacy integrations registry with smsnetbd, bdbulksms, mimsms, and gennet. Delivery uses legacy factory/provider files; universal delivery provider exports are type-only.
- **In-memory state**: Storefront L1 caches and shared layout cache are in-memory and reset on Worker isolate restart. Generic low-risk throttles may still use the shared KV rate limiter, but auth, setup, checkout, payment, and inventory authorities must use D1/DO/queue-backed coordination instead of KV get-then-put.
- **Notification/OTP delivery receipts**: Order notifications and customer OTP delivery now have D1 delivery receipts, Resend idempotency keys, deterministic GenNet `csms_id`, Meta WhatsApp message-id capture, and provider receipt recording. Remaining work: revisit Cloudflare Email/FCM/WhatsApp if first-class provider idempotency keys become available.
- **Scanner app**: Standalone `/scanner` route exists with D1 single-use QR-token auth and KV scanner sessions; still needs focused UI polish beyond the auth exchange.
- **Type safety**: Admin still has notable `any` usage in some UI edges and broad DTO adapters. Improve locally when touching relevant code, but do not broad-refactor casually.
- **Test coverage**: There are dozens of Vitest files across apps/packages, not a comprehensive suite. Add focused tests for risky service, queue, payment, inventory, response-envelope, AI, or storefront cache changes.
- **Generated docs are non-authoritative**: Trust `packages/api-client/openapi.json`, generated SDK files, Drizzle schema, and migration metadata over prose counts.

## Agent Team Guidelines

When working as part of an agent team on this codebase:

- **Avoid file conflicts**: coordinate so each teammate owns different files/modules.
- **Domain boundaries**: `packages/core/src/modules/` is organized by domain; each teammate should own a complete domain when possible.
- **Run type checks**: use `pnpm typecheck` for meaningful validation. `pnpm build` bundles Workers/apps and can miss type errors that `tsc`/`astro check` catch.
- **Run lint honestly**: root `pnpm lint` intentionally filters out `@scalius/tsconfig` and runs real `lint` scripts for API, admin, storefront, api-client, core, database, and shared. Keep it warning-free; if a future warning is truly package/toolchain-dependent, document the exact reason instead of normalizing warning noise.
- **Run focused tests**: use `pnpm test` or direct Vitest filters for touched areas.
- **Run dependency audit**: use `pnpm audit --audit-level moderate` after dependency or lockfile changes.
- **Do not touch env files**: `.dev.vars` and `.env.development` can contain secrets.
- **Keep Turbo env cache inputs current**: app-local `.dev.vars`/`.env*` files belong in `turbo.json` `globalDependencies`, and build-time env names belong in `globalEnv`. Do not add secret-only names to `globalEnv`. Run `pnpm exec vitest run scripts/turbo-config.test.mjs` and a Turbo dry run after changing build env/config.
- **Schema changes need migrations**: after modifying `packages/database/src/schema/`, run `pnpm db:generate`.
- **API surface changes need SDK regeneration**: after changing OpenAPI route schemas, run `pnpm generate:sdk`.
- **Package changes need manifests**: when adding imports from a new package, update the consuming workspace `package.json`.
- **Storefront shared imports only**: do not add `@scalius/core` or `@scalius/database` imports to storefront without coordination.
- **Generated files are off-limits**: do not hand-edit `routeTree.gen.ts` or `packages/api-client/src/generated/**`.
- **Cloudflare bindings must stay synchronized**: update Wrangler config and Env declarations together.
- **Deploy shortcuts stay safety-gated**: root and package-local deploy scripts route through `scripts/deploy.mjs --only ...`; use `pnpm run deploy*` from the root to avoid pnpm's built-in `deploy` command, and keep typecheck, dist-secret checks, and required migration gates when changing deploy scripts.
